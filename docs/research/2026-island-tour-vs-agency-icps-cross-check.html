🔴 ISSUE 2: Bokun Mobile App Mischaracterized as Consumer-Facing

The report claims Bokun has a consumer-facing "Bókun Mobile App" [r1] alongside Travefy and Scott Dunn's client apps. Bokun's mobile app is actually operator/supplier-facing (ticket scanning, booking management, check-ins) — it is B2B, not a traveler-facing client app.

Source: bokun.io, Capterra review

Fix: Remove Bokun from the "Mobile client app" row in the shared features table, or clarify it's an operator app, not a client portal app.

🟡 ISSUE 3: "60/40 shared/different" Ratio is an Author Estimate

The TL;DR states agencies "share roughly 60% of their data model, workflows, and feature surface" with "40% that differs." No industry benchmark or analyst source exists for this ratio. It appears to be a reasonable heuristic derived from the feature matrix, but it should be labeled as the author's estimate, not presented as an established figure.

🟡 ISSUE 4: Fora Commission Model Oversimplified

The report claims advisors route "commission across 3-5+ suppliers, with split rules per supplier contract." In reality, a single booking typically involves one supplier. The commission split is between Fora (host) and the advisor (typically 70/30), not fragmented across multiple suppliers per booking. The advisor books multiple suppliers across a trip, but commission flows per-booking, not per-trip.

Source: Fora: How Do Travel Agents Get Paid

Advisors don't handle client money — confirmed. "Dual-entry payment ledger" is an accounting practice, not a regulatory requirement — the report shouldn't present it as regulatory mandate.

The report's "20% / 60-90 days" generalizes the group rate to all bookings.

5. Recommended Corrections

Must Fix

  1. Realign reference numbering. The Sources list [r4]–[r9] doesn't match body text citations. Either renumber the Sources list to match the body, or update every in-text citation. The body text appears to be the "correct" intent.
  2. Correct Bokun mobile app characterization. Remove Bokun from the "Mobile client app" shared-feature row or add a note that it's operator-facing, not consumer-facing.
  3. Fix Fora commission model description. Change "commission splits across 3-5+ suppliers" to "commission flows per-booking from supplier → host agency → advisor (typically 70/30 split)." Multiple suppliers exist per trip, not per booking.

Should Fix

  1. Label "60/40" as an estimate. Add "estimated" or "approximately, based on the feature matrix below" before the 60/40 claim.
  2. Soften safari deposit terms. Change "20% deposit, balance 60-90 days" to "10-20% deposit, balance 45-90 days depending on group size" per &Beyond's actual terms.
  3. Clarify ATOL/ABTA "dual-entry." Trust fund segregation is regulatory; "dual-entry payment ledger" is an accounting practice, not a regulatory requirement.
  4. Downgrade Honeyfund/Zola from "integration" to "workflow compatibility." No formal API integration exists.
  5. Add FareHarbor [r5-sources] and Checkfront [r6-sources] to the body text or remove from Sources list — they're currently orphaned.
  6. Add TravelJoy reference to the body text where it's mentioned ("Travefy and TravelJoy have this") — currently unreferenced inline.
